some pointers in taste

Publish date: Nov 7, 2020
Tags:

Readers be warned! This is more technical than other posts of mine. I’ve been helping a friend learn some elements of programming, and the topic of memory came up.

It may take a second read, but I think it’s well worth it!

In 2016 Linus Torvalds, who created Linux (the OS most of the internet runs on) and git (the content management system this blog uses), sat down for with Chris Anderson, who is in charge of Ted, for a chat. The full interview is about 22 minutes long. If you have the time, it’s well worth watching.

During the interview, Chris asks Linas about his concept of “taste” (at 14:20, video will automatically start from here). Linas had brought two code snippets, one that was bad “taste” and one that was good “taste”. He explained the difference between the two,

Sometimes you can see a problem in a different way and re-write it so that a special case goes away and becomes the normal case. And that’s good code

For me, this is the special sauce. Whatever reasons one is drawn to computer science, this is the reason they stay. There is an unparalleled joy in understanding an elegant idea. By the end of this blog, you’ll have the palette to understand why the second example is in better taste, and I hope you’ll derive some joy from it like the rest of us nerds.

memory

Before we can appreciate the differences between these approaches (and also what’s going on), we need a working understanding of computer memory. Consider a row of lockers – like the ones you had at school – laid out in a line.

Each locker contains one letter or number. We’ll label that data red. Each locker also has an index, so we can tell one locker from another. We’ll label that blue.

The first piece of data stored is “h”. There are two ways refer to it. The first is directly. We explicitly talk about the letter “h”. The second way is indirectly. We can talk about “the letter in locker 0”. They both refer to the same thing, but in two different ways.

This second idea is why we call them pointers – they point to the place in memory where data is stored, rather than the data itself.

In C (the programming language in these examples), this different way of addressing things looks like this,

/* C-ing clearly! */

/* Keeping track of something directly */ 
char direct = "a";

/* Keeping track of something indirectly */
char* indirect = &direct;

For those unfamiliar with C this can look like, well, a different language. Let’s unpack.

  • char corresponds to the type of data being stored. It’s short for “character”. Different data types are classified differently. For example, int for an integer. Keeping track of data’s type helps make sure we don’t accidentally use a number when we actually need a letter.
  • char* corresponds to a pointer to char-type data.
  • & is the symbol C uses when you want to get the pointer to some data, rather than the data itself. In this case &direct is the address of direct. It’s the number of the locker that holds that data.

linked lists

We often want to create more interesting ways of storing data than this dull row of blocks. But that is what we have to work with. If we want to do something else, we must overlay it on top of what we have. One of those more interesting structures is a linked list. The idea is we want to store some data as a collection of elements, where each element links to the next one. Here’s a picture

We can implement this by storing both the element in the list, and the link to the next element, in memory.

Then all we need to know is the address of the first element, and we can go forward in the list from there. This being said, what’s next?

first method

We are now equipped to understand both methods, and the aesthetic differences between the two. The first method is the ‘bad taste’ version Linus was referring to. Here’s the code for it.

remove_list_entry(entry)
{
	prev = NULL;
	walk = head;
	
	// Walk the list
	while (walk != entry)
	{
		prev = walk;
		walk = walk->next;
	}
	
	// Remove the entry by updating the head
	// or previous entry
	
	if (!prev)
		head = entry->next;
	else
		prev->next = entry->next;
}

This is the “natural” way to remove a list entry. We start with the first entry in the linked list, referred to as the head. Then we walk the list, checking each entry in turn until we have a match between the entry we are looking at in the walk, and the entry we want to delete (entry). (I have not explained this well).

There is a special case we must consider. What if the entry we want to delete is the first entry – the head?

second method

Now, the second method. The one with good taste. This is going to be hard to explain. I feel like this won’t really help anyone. Although then again, it is the blog I would have liked to read when I was 18

remove_list_entry(entry)
{
	// The "indirect" pointer points to the
	// *address* of the thing we'll update
	
	indirect = &head;
	
	// Walk the list, looking for the thing that 
	// points to the entry we want to remove
	
	while ((*indirect)) != entry)
		indirect = &(*indirect)->next;
		
	*indirect = entry->next;
}

Get the pointer to the pointer of the head. Fucking magic! Then we find the pointer we want to remove, and just set it to the next one.
